# T.A.C.I.D. — Канальная скорость, параметры связи и предельные режимы

Документ оставлен только по теме пропускной способности и параметров канала связи с учетом примененных мер защиты.

Цель: показать реальную и предельную скорость передачи данных для текущей архитектуры, а также границы применимости подхода при переносе на другие системы (включая видеопоток).

---

## 1. Что означает req/s

`req/s` (requests per second) — количество HTTP-запросов в секунду, которое обрабатывает серверный endpoint телеметрии.

Пересчет в скорость данных:

- `R_req = N / T`
- `R_L7_Bps = R_req * S_payload`
- `R_L7_bps = R_L7_Bps * 8`

Где:
- `S_payload` — размер одного telemetry-пакета на уровне приложения (JSON + поля защиты).

Фактический размер benchmark-пакета в текущем проекте:

- `S_payload = 344` байта

---

## 2. Параметры канала связи и защиты (влияющие на скорость)

Канал маяк -> сервер:
- транспорт: HTTP(S)
- формат: JSON
- защита: HMAC-SHA256, replay-check (sequence/timestamp), rate limiting

Параметры rate limiting в сервере:
- Redis token bucket: `capacity = 20`, `refill = 15 token/s` на ключ `(beaconId, sourceIp)`
- fallback (без Redis): до `20 req/s` на `(beaconId, sourceIp)` за окно 1 сек

Практический смысл:
- при включенном Redis устойчивый разрешенный поток от одного маяка ограничен примерно `15 req/s`;
- в fallback-режиме — примерно `20 req/s`.

---

## 3. Фактические измерения на текущем стенде

Источник: security/mixed benchmark (8 секунд, локальный dev-стенд).

Измерено:
- `legitTotal = 6`, `legitBlocked429 = 0`
- `floodTotal = 122`, `floodBlocked429 = 40`

### 3.1 Скорость по каналам (L7 payload)

| Канал | N | T, c | Скорость, req/s | Скорость L7, кбит/с |
|---|---:|---:|---:|---:|
| Легитимный поток (до защит) | 6 | 8 | 0.75 | 2.06 |
| Легитимный поток (после защит) | 6 | 8 | 0.75 | 2.06 |
| Flood поток (до защит) | 122 | 8 | 15.25 | 41.97 |
| Flood поток (после защит) | 82 | 8 | 10.25 | 28.21 |
| Отсечено защитами (429) | 40 | 8 | 5.00 | 13.76 |
| Суммарно (до защит) | 128 | 8 | 16.00 | 44.03 |
| Суммарно (после защит) | 88 | 8 | 11.00 | 30.27 |

Вывод по факту:
- защита существенно режет flood-поток;
- легитимный поток в этом сценарии не просаживается (`FalsePositiveRate = 0%`).

## 4. Реально достижимые максимальные скорости

Ниже — предельные значения для текущей схемы (JSON + текущий limiter), без изменения протокола.

### 4.1 Теоретический максимум на 1 маяк

При Redis limiter (`15 req/s`):

- `R_L7 = 15 * 344 = 5160 B/s = 41.28 кбит/с (~5.04 КиБ/с)`

При fallback limiter (`20 req/s`):

- `R_L7 = 20 * 344 = 6880 B/s = 55.04 кбит/с (~6.72 КиБ/с)`

### 4.2 Масштабирование по числу маяков

Если все маяки независимы по `(beaconId, sourceIp)`, L7-скорость суммируется почти линейно:

- Redis режим: `R_total_L7 ~= N_beacons * 41.28 кбит/с`
- Fallback режим: `R_total_L7 ~= N_beacons * 55.04 кбит/с`

Пример (Redis, 15 req/s):

| Маяков | Оценка L7, кбит/с | Оценка L7, МБ/с |
|---:|---:|---:|
| 10 | 412.8 | 0.052 |
| 50 | 2064.0 | 0.258 |
| 100 | 4128.0 | 0.516 |
| 200 | 8256.0 | 1.032 |
| 400 | 16512.0 | 2.064 |

### 4.3 Оценка L4/L3 (с HTTP/TCP/TLS overhead)

Для коротких JSON-запросов типично, что полезная доля L7 составляет примерно 35-55% от «провода» (зависит от TLS, keep-alive, MTU, RTT).

Отсюда грубая оценка:

- `R_link ~= R_L7 / eta`, где `eta = 0.35..0.55`

Для цели L7 = 1.0 МБ/с:
- `R_link ~= 1.82..2.86 МБ/с`

Для цели L7 = 2.0 МБ/с:
- `R_link ~= 3.64..5.71 МБ/с`

## 5. Можно ли получить 1-2 МБ/с на текущих настройках

Короткий ответ:
- для типичного режима T.A.C.I.D. (5-10 Гц и небольшие пакеты телеметрии) текущие настройки подходят и рациональны;
- для задач типа видеопотока текущая схема передачи (малые JSON-запросы + строгий per-beacon limiter) не является эффективной.

Расчет по текущей схеме (Redis, 15 req/s, 344 байта):

- для `1 МБ/с` L7 нужно примерно `~2907 req/s` суммарно
- для `2 МБ/с` L7 нужно примерно `~5814 req/s` суммарно

Эквивалент по маякам при 15 req/s на маяк:

- `~194 маяка` для 1 МБ/с
- `~388 маяков` для 2 МБ/с

Это уже другой класс нагрузки и архитектуры.

## 6. Решение по настройкам

### 6.1 Для T.A.C.I.D. (позиционирование игроков)

Текущие настройки оставляем:
- они обеспечивают хороший баланс безопасности и устойчивости;
- limiter эффективно срезает flood при сохранении легитимного трафика.

### 6.2 Для переноса подхода на высокоскоростные системы (например, видео)

Сохранять только «как есть» нельзя; нужно адаптировать транспортный слой:

1. Бинарный формат вместо JSON (MessagePack/Protobuf).
2. Батч-передача (несколько измерений/кадров в одном сообщении).
3. Перенос потоковых данных на протоколы, подходящие под media streaming (RTP/QUIC/WebRTC), а HMAC/replay/rate-limit применять на управляющем/сигнальном канале.
4. Профиль limiter для доверенных high-throughput источников (отдельные квоты, не 15 req/s).
5. Обязательные стендовые замеры L7 и L4/L3 под целевой bitrate.

---

## 7. Итог

- `req/s` — это частота запросов, а не напрямую «скорость в МБ/с».
- В текущем проекте при `S_payload = 344` байта и текущих лимитах скорость одного маяка ограничена примерно `41.28-55.04 кбит/с`.
- Для целей позиционирования это достаточно и корректно.
- Для видеопотока 1-2 МБ/с подход по безопасности переносим, но транспорт и параметры канала требуется проектировать отдельно.
